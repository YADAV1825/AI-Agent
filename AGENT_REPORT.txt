1. EXECUTIVE SUMMARY
================================================================================

The Opus Desktop AI Agent is a sophisticated MVP platform that transforms any API 
key (Lightning AI / Ollama compatible models) into an autonomous OS-level intelligent 
agent capable of:

- Complete desktop automation through keyboard, mouse, and shell command execution
- Self-healing and self-correcting behavior through visual feedback loops
- Multi-modal task execution (visual analysis, code execution, file operations)
- Human-in-the-loop safety mechanisms with user confirmation gates
- Voice-based task initiation (optional, gracefully falls back to text)
- Real-time visual verification through automated screenshot analysis
- Persistent execution logging and history tracking
- Reflex command optimization for immediate task execution

The system bridges the gap between API-driven LLM services and direct operating 
system control, creating an autonomous agent that can perform complex multi-step 
tasks with minimal human intervention.

3.2 SELF-HEALING THROUGH VISUAL FEEDBACK
------------------------------------------

The agent implements a sophisticated "See-Think-Act-Verify" loop:

1. SCREENSHOT ANALYSIS
   - Captures current screen state before actions
   - Uses vision_analysis field to describe screen context
   - Identifies UI elements, buttons, text, and content

2. INTELLIGENT ACTION SELECTION
   - Chooses appropriate keyboard shortcuts or mouse actions
   - Uses tab navigation to focus target elements
   - Adapts based on visual feedback from previous steps

3. VERIFICATION STEP
   - Takes follow-up screenshots to confirm action success
   - Compares expected vs. actual state
   - Automatically retries with alternative approaches if action fails

4. ADAPTIVE RECOVERY
   - If action fails, agent tries different methods
   - Can switch between clicking, typing, keyboard shortcuts
   - Learns from failure patterns within same execution

EXAMPLE FLOW (Playing a song on YouTube):
  Step 1: take_screenshot() → "I see YouTube homepage"
  Step 2: press_key("ctrl+l") → Focus address bar
  Step 3: type_text("youtube.com/search?q=Song") → Search
  Step 4: press_key("enter") → Navigate
  Step 5: take_screenshot() → "I see search results with first video"
  Step 6: press_key("tab") → Navigate to play button
  Step 7: press_key("enter") → Click play
  Step 8: take_screenshot() → Verify video is playing with progress bar


3.3 ADVANCED COMMAND MODES
---------------------------

The agent supports 5 distinct command interpretation modes:

MODE 1: REFLEX COMMANDS (Immediate Direct Actions)
  - Direct user commands bypass planning
  - Examples: "press alt tab", "type hello", "hit enter"
  - Skips plan.txt creation
  - Response time: <1 second
  - Used for: Quick, immediate actions

MODE 2: SCREEN-CONTEXT COMMANDS (Visual Awareness)
  - Commands referencing visible screen content
  - Examples: "play the song you see", "click that button"
  - Mandatory screenshot as first action
  - Full vision_analysis field required
  - Self-healing via visual feedback

MODE 3: MULTI-STEP PLANNING (Complex Tasks)
  - Traditional task execution with numbered steps
  - Creates plan.txt with execution strategy
  - Updates plan with [DONE] markers
  - Used for: Complex multi-step workflows
  - Example: "Open YouTube and search for Faded and play it"

MODE 4: EMAIL COMPOSITION (Gmail Specific)
  - Dedicated mode for email operations
  - Detects account index (/u/0/, /u/1/)
  - Fills To, Subject, Body using keyboard
  - Uses Ctrl+Enter for sending
  - Verifies with screenshot

MODE 5: WINDOW SWITCHING (Alt-Tab Navigation)
  - Advanced Task Switcher menu navigation
  - Multi-step process with visual verification
  - Holds Alt key across multiple steps
  - Captures target window selection
  - Confirms final window focus

→ System automatically detects which mode to use based on task format


3.4 SECURITY & SAFETY MECHANISMS
---------------------------------

BUILT-IN SAFETY GATES:

1. USER CONFIRMATION FOR DANGEROUS OPERATIONS
   - Shell command execution: Shows command preview, requires user approval
   - Python code execution: Shows code snippet, requires confirmation
   - Prevents: Accidental system damage, malware-like behavior

2. 40-STEP SAFETY THRESHOLD
   - After 40 steps, agent pauses and asks user
   - Dialog: "I have reached 40 steps. Should I keep going?"
   - User can stop at any milestone
   - Prevents: Infinite loops, runaway automation
   - Timeout: 120 seconds for user response

3. STOP REQUEST HANDLING
   - User can stop agent at any time (⏹ button)
   - Agent gracefully handles InterruptedError
   - Browser kept open for manual intervention
   - Process and file states preserved

4. COMMAND VALIDATION
   - JSON output strictly validated
   - Malformed responses detected and logged
   - Timeouts on subprocess operations (90 seconds for shell, 60 for Python)
   - Keyboard input sanitization (converts aliases to standard key names)

5. VISION-BASED VERIFICATION
   - Actions verified through screenshot analysis
   - Prevents blind command execution
   - Automatic fallback if action doesn't produce expected result

6. HEADLESS FALLBACK
   - If no GUI root available, auto-continues past 40-step check
   - Used for server deployments or batch operations
   - Returns True by default to keep automation running

4.1 CORE/AGENT.PY - ORCHESTRATION ENGINE
-----------------------------------------

DesktopAgent Class:
  └─ Primary orchestrator for the autonomous execution loop

Key Methods:

1. _is_direct_command(task)
   - Detects reflex and screen-context commands
   - Checks against predefined prefixes: "press ", "click ", "type ", etc.
   - Checks screen-context keywords: "you see", "on the screen", etc.
   - Returns: Boolean to skip plan.txt

2. _make_messages(task, history, enforce_plan, latest_result)
   - Constructs conversation messages for LLM
   - Compact history: Last 8 steps (limited to prevent token bloat)
   - Includes vision_analysis for screenshot understanding
   - Injects base64-encoded image data if screenshot was taken
   - Returns: [SystemPrompt, UserMessage]

3. _call_planner(task, history, enforce_plan, latest_result)
   - Calls LLM to get next action (JSON)
   - Extracts first JSON block from response
   - Validates required fields
   - Returns: (parsed_dict, raw_response)

4. _ask_continue(turn)
   - Thread-safe dialog box after 40 steps
   - Spawns in main Tk thread via .after(0, _show_dialog)
   - Waits up to 120 seconds for user response
   - Returns: Boolean (continue/stop)

5. run_task(state, step_callback)
   - Main execution loop
   - Logs task and model info
   - Repeats: Plan → Execute → Verify → Update History
   - Safety check every 40 steps
   - Enforces plan.txt creation on first step (unless reflex)
   - Catches all exceptions and logs errors
   - Keeps browser open for user continuation

Execution Loop Constants:
  - SAFETY_THRESHOLD = 40 steps
  - Subprocess timeouts: 90 seconds (shell), 60 seconds (Python)
  - Step callback for UI updates
  - Compact history window: 8 steps

4.5 TOOLS/OS_TOOLS.PY - SYSTEM OPERATIONS
------------------------------------------

OSTools Class: Desktop control interface

KEY METHODS:

1. press_key(key)
   - Single key: "enter", "tab", "space", "f5"
   - Simple combo: "ctrl+l", "ctrl+shift+esc"
   - Modifier hold: "alt+tab" (holds, presses, releases)
   - Repeated: "alt+tab+tab" (holds alt, presses tab twice)
   - Special handling for win, winleft, winright keys
   - Sleep timings for OS stability (1s after win, 0.5s after alt)
   - Returns: Action description

2. type_text(text)
   - Uses pyautogui.write() with 0.01s interval
   - Safe character input to focused window
   - No special character interpretation
   - Simple, reliable method

3. take_screenshot()
   - PIL ImageGrab for Windows/Mac compatibility
   - Saves as: temp_screenshot.jpg (JPEG quality 85)
   - Returns: "[SCREENSHOT_TAKEN: /abs/path]" format
   - Used for vision analysis in next step

4. open_app(app_name)
   - Special handling for Chrome (guest mode option)
   - VS Code: subprocess.Popen("code")
   - Notepad: subprocess.Popen("notepad")
   - Generic: subprocess.Popen(app_name, shell=True)
   - Cross-platform: Windows, macOS, Linux support

5. open_url(url)
   - Uses webbrowser.open() for default browser
   - Respects user's browser preference
   - Opens in new tab or window

6. search_web(query)
   - Constructs Google Search URL
   - Encodes query with requests.utils.quote()
   - Opens in default browser

7. read_url(url, max_chars=12000)
   - HTTP GET with timeout (25 seconds)
   - BeautifulSoup HTML cleaning (if available)
   - Removes script, style, noscript tags
   - Compresses whitespace
   - Truncates at 12KB
   - User-Agent header included

8. shell(command, confirm_fn)
   - User confirmation gate (shows command preview)
   - subprocess.Popen with shell=True
   - 90-second timeout
   - Stop check for clean interruption
   - Captures stdout and stderr
   - Returns exit code and output


4.6 TOOLS/FILE_TOOLS.PY - FILE OPERATIONS
------------------------------------------

FileTools Class: File system interface

Methods:

1. read_file(path, max_chars=8000)
   - Uses pathlib.Path for cross-platform paths
   - Expanduser (~) and resolve()
   - Returns first 8KB (truncated if larger)
   - Error handling for missing files

2. write_file(path, content)
   - Creates parent directories if needed
   - Overwrites existing files
   - UTF-8 encoding
   - Returns absolute path

3. append_file(path, content)
   - Opens in append mode
   - Creates parent directories
   - UTF-8 encoding

4. list_dir(path)
   - Returns sorted directory listing
   - Directories marked with trailing "/"
   - Returns file content if path is file (first 4KB)
   - Sorts: directories first, then files

6. SECURITY & SAFETY ANALYSIS
================================================================================

6.1 THREAT MODEL MITIGATION
----------------------------

THREAT 1: Malicious shell commands
  ├─ MITIGATION: User confirmation gate on all shell() calls
  ├─ PREVIEW: Command preview (150 chars) shown before execution
  ├─ TIMEOUT: 90-second hard limit prevents long-running exploits
  └─ AUDIT: Full command logged in execution history

THREAT 2: Runaway infinite loops
  ├─ MITIGATION: 40-step safety threshold
  ├─ USER CHECK: Dialog every 40 steps asking to continue
  ├─ TIMEOUT: 120-second timeout for user response
  ├─ EARLY STOP: "done" action checked immediately
  └─ LOG: Step count tracked and reported

THREAT 3: Accidental file deletion
  ├─ MITIGATION: No automatic deletion (write, append, read only)
  ├─ SCOPE: File operations limited to user directories
  ├─ VISIBILITY: All file operations logged
  └─ VERIFICATION: No wildcard deletion patterns

THREAT 4: Unauthorized code execution
  ├─ MITIGATION: run_python requires user confirmation
  ├─ PREVIEW: Code snippet preview (200 chars) shown
  ├─ TIMEOUT: 60-second limit on Python execution
  ├─ SANDBOX: Subprocess isolation
  └─ OUTPUT: Stderr/stdout captured separately

THREAT 5: Browser-based CSRF/XSS
  ├─ MITIGATION: Browser tools optional (graceful fallback)
  ├─ SCOPE: Limited to browser automation, no DOM access
  ├─ ISOLATION: Separate browser instance
  └─ LOG: All browser actions logged

THREAT 6: Stop request not working
  ├─ MITIGATION: Stop check on every iteration
  ├─ SUBPROCESS: Process.kill() on stop
  ├─ LLM: Response stream interrupted
  └─ FEEDBACK: InterruptedError caught and logged


6.2 PERMISSION MODEL
--------------------

API KEY ACCESS
  - Requires: LIGHTNING_API_KEY environment variable
  - Authentication: Bearer token + teamspace
  - Scope: API calls only (cannot access other services)
  - Rotation: Managed externally (not stored in code)

FILE SYSTEM ACCESS
  - Read: Any file, location unrestricted
  - Write: Only via write_file/append_file actions
  - Delete: Not supported (design choice)
  - Hidden files: Accessible but not recommended

KEYBOARD/MOUSE ACCESS
  - Unrestricted within OS limitations
  - No special privileges needed
  - Subject to OS focus/foreground restrictions

SHELL ACCESS
  - Requires: User confirmation before each command
  - Restrictions: User-level processes only
  - Privileged commands: May fail due to UAC/permissions


6.3 PRIVACY CONSIDERATIONS
--------------------------

DATA COLLECTION
  - Screenshots sent to LLM for vision analysis
  - Contains: Visual content of screen (potentially sensitive)
  - Recommendation: Review screenshots for sensitive data
  - Control: User can review all logs before they're sent

HISTORY LOGGING
  - All steps logged to disk in ./logs/ directory
  - Contains: Commands, outputs, decisions
  - Retention: User-controlled (can be deleted)
  - Format: Plain text, human-readable

NETWORK TRANSMISSION
  - API calls to Lightning AI servers
  - HTTPS encryption used
  - Payload: LLM messages and screenshots
  - Risk: Depends on Lightning AI's privacy policy

RECOMMENDATIONS
  - Review sensitive data before screenshots
  - Use local Ollama client for sensitive tasks
  - Delete logs containing sensitive information
  - Use VPN/proxy if on public networks

9. FAILURE MODES & RECOVERY
================================================================================

COMMON FAILURE POINTS:

1. LLM API Timeout
   ├─ Symptom: "timeout during model response"
   ├─ Recovery: Automatic retry (max 2 attempts)
   ├─ Mitigation: Increase timeout (currently 120s)
   └─ User Action: Stop and restart if persistent

2. JSON Parse Failure
   ├─ Symptom: "Could not parse JSON from model response"
   ├─ Recovery: extract_first_json_block with fallback matching
   ├─ Mitigation: Improved prompt engineering
   └─ User Action: Review raw response in logs

3. Keyboard Input Not Registered
   ├─ Symptom: Text not appearing in window
   ├─ Recovery: Automatic screenshot → retry decision
   ├─ Mitigation: Focus window first, add delays
   └─ User Action: Click target window manually, restart task

4. Screenshot Capture Failure
   ├─ Symptom: "[SCREENSHOT_TAKEN: ... ] but file missing"
   ├─ Recovery: Error logged, task continues with "blind" next step
   ├─ Mitigation: Fall back to previous screenshot
   └─ User Action: Manual verification may be needed

5. Shell Command Timeout
   ├─ Symptom: "subprocess.TimeoutExpired after 90 seconds"
   ├─ Recovery: Process killed, error returned
   ├─ Mitigation: Increase timeout or optimize command
   └─ User Action: Use faster command or break into steps

6. File Not Found
   ├─ Symptom: "File not found: /path/to/file"
   ├─ Recovery: Error logged, agent can retry with different path
   ├─ Mitigation: Always use absolute paths
   └─ User Action: Provide correct file path

7. Window Focus Lost
   ├─ Symptom: Typing/clicking goes to wrong window
   ├─ Recovery: yield_focus() tries to minimize first
   ├─ Mitigation: Take screenshot first to verify focus
   └─ User Action: Click target window or restart task

8. Stop Request Not Honored
   ├─ Symptom: Agent continues after stop button pressed
   ├─ Recovery: Hard timeout on next subprocess call
   ├─ Mitigation: Check stop_check more frequently
   └─ User Action: Force kill process if necessary


SELF-HEALING STRATEGIES:

The agent employs several automatic recovery techniques:

1. SCREENSHOT-BASED DECISION MAKING
   - Takes screenshot before critical actions
   - Uses vision_analysis to understand current state
   - Automatically adapts if screen state differs from expectation

2. ALTERNATIVE ACTION SELECTION
   - If keyboard input fails, tries click + tab approach
   - If direct URL navigation fails, uses search method
   - If button click fails, tries keyboard shortcut

3. HISTORY-BASED CONTEXT
   - Maintains last 8 steps in history
   - Learns from previous step outcomes
   - Avoids repeating failed actions exactly

4. ADAPTIVE TIMING
   - Adds delays based on operation type (1.5s for screenshot)
   - Waits after system keys (win: 1s, alt: 0.5s)
   - Adjustable timeouts in configuration

5. FALLBACK MODES
   - Ollama fallback if Lightning API fails
   - GUI fallback if voice recognition unavailable
   - Text entry fallback if voice fails
   - Headless execution mode available
